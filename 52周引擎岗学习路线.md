# 52 周引擎岗学习路线（纯内容版）

> 适用对象：计算机专业研究生 / 具备一定 C++ 基础 / 目标为游戏引擎岗、渲染岗或图形基础设施方向。  
> 文档目标：形成一份可直接执行、可持续产出项目、可服务求职的全年学习路线。  
> 文档范围：覆盖 **52 周引擎岗学习路线 + 学习资源 + 每周任务 + 阶段目标 + 验收标准 + 求职导向产出**。

---

## 一、路线定位与总体目标

这份路线的核心不是“把知识点看完”，而是用 52 周把能力沉淀为能够在校招、实习或社招初中级引擎岗中展示的成果。整个路线围绕以下三条主线并行推进：

### 主线 A：基础能力线
全年持续推进，目标是形成稳定的底层能力。

包含：
- C++ 工程能力
- 数据结构与算法
- 操作系统与计算机体系结构
- 图形学数学基础
- 实时渲染核心理论
- 性能分析与调试能力

### 主线 B：项目作品线
每个阶段都必须形成可运行、可展示、可写进简历的产出。

建议最终沉淀 4 个核心项目：
1. 高性能基础组件库（Allocator / ThreadPool / TaskGraph / Benchmark）
2. 3D Math + Collision 模块
3. Software Renderer
4. GPU Mini Engine / Renderer

### 主线 C：求职表达线
从第 1 周开始积累，而不是最后突击。

持续产出：
- 技术笔记
- 项目 README
- 架构图和流程图
- 性能优化记录
- 面试题卡片
- 模拟面试答案库

## 三、全年学习原则

## 1. 以项目驱动学习
每一个阶段都必须收敛到可运行项目，而不是停留在“看过教程”和“记过笔记”。

## 2. 以可解释性为目标
每学完一个模块，必须能够回答：
- 为什么这样设计？
- 有哪些替代方案？
- 当前实现的性能瓶颈在哪里？
- 如果重做一遍，会怎么改？

## 3. 先完整，再复杂
所有项目都按以下四层推进：
1. 最小可运行版本
2. 功能完整版本
3. 调试可视化版本
4. 性能优化版本

## 4. 面试准备全年滚动进行
从第 1 周开始：
- 每周 3 到 5 道算法题
- 每周整理 5 个面试问答
- 每月进行 1 次项目复盘或模拟面试

## 5. 所有阶段都必须留档
每个阶段结束时至少保留：
- README
- 项目截图或录屏
- 架构图
- 关键流程图
- 一页性能或问题复盘

---

## 四、推荐学习节奏

- 3 天项目开发

## 五、核心资源使用方式

下面不是单纯的书单，而是“怎么用这些资料”。

### 1. C++ / 工程基础
- `cppreference`：查语法、标准库、并发库、容器、算法接口。不要把它当教程，而是当权威字典。
- 《Effective Modern C++》：用于重构现代 C++ 习惯和资源管理意识。
- 《C++ Concurrency in Action》：用于线程、同步、并发模型的系统学习。
- 《Game Engine Architecture》：用于建立“引擎不是只有渲染”的全局工程认知。
- CMake 官方 Tutorial：完整走一遍，掌握 target、include、test、install 的基本用法。

### 2. 数学 / 图形学基础
- GAMES101：建立图形学整体框架，尤其是坐标变换、光栅化、基础光照。
- GAMES102：补几何基础、曲线曲面与图形数学认知。
- GAMES103：作为动画与物理的补充认知。
- Scratchapixel：补齐“为什么这样算”的几何和渲染底层直觉。
- 《Mathematics for 3D Game Programming and Computer Graphics》：系统化整理数学基础。

### 3. 实时渲染 / GPU 渲染工程
- LearnOpenGL：作为软渲染器到 GPU 渲染框架之间的工程过渡。
- Vulkan Tutorial：用于现代 API 的第一套系统实战路线。
- 《Real-Time Rendering》：作为实时渲染体系的中长期参考书。
- 《Physically Based Rendering（PBRT）》：作为 PBR、材质、采样与高级渲染原理的跳读资料。

### 4. 工具链与调试
- RenderDoc：抓帧分析渲染过程、pass、draw call、attachment 与资源状态。
- Tracy：做 CPU 侧性能分析，观察线程、区域耗时与帧级热点。
- AddressSanitizer / UBSan / ThreadSanitizer：用于内存越界、未定义行为和数据竞争排查。
- gdb / lldb：用于底层调试。

### 5. 算法与面试
- LeetCode：按专题刷，不追求题海。
- 面试卡片：每周整理 C++、图形学、OS、项目表达各类问答。

# 阶段一：C++ 与系统工程基础（第 1-10 周）

## 阶段目标
把“会写 C++”提升为“能写引擎风格 C++ 工程代码”。

## 阶段产出
- CMake 多模块项目骨架
- 日志、断言、测试、benchmark 基础设施
- Allocator 组件
- Thread Pool
- TaskGraph / Job System v1
- 基础性能测试报告

## 第 1 周：环境、仓库与工程骨架搭建
### 学什么
- Git 工作流
- CMake 基础结构
- 编译器警告级别
- 日志系统、断言系统、简单单元测试

### 怎么学
- 按 CMake 官方 Tutorial 搭建最小工程
- 建立统一 monorepo：
  - `engine_core`
  - `math`
  - `renderer_cpu`
  - `renderer_gpu`
  - `tools`
  - `tests`
- 配置 Debug / Release
- 接入简单测试框架与 benchmark 框架

### 本周任务
- 创建工程目录与根 CMakeLists
- 写一个 Logger 和 Assert 宏
- 写一个 Hello Engine 程序并跑通测试
- README 中记录目录结构与构建方法

### 本周验收
- 能一键构建 Debug / Release
- 能运行单元测试
- 能说清楚工程分层目的

---

## 第 2 周：现代 C++ 基础重整
### 学什么
- RAII
- move semantics
- perfect forwarding
- smart pointer 使用边界
- 值类别和对象生命周期

### 怎么学
- 对照 cppreference 查阅特性和示例
- 用小实验验证拷贝、移动、临时对象和析构时机
- 阅读《Effective Modern C++》中与资源管理相关章节

### 本周任务
- 实现文件句柄封装类
- 实现计时器类和 ScopeTimer
- 实现不可拷贝但可移动的资源类
- 记录常见错误：悬空引用、重复释放、过度使用 shared_ptr

### 本周验收
- 能解释何时该用 unique_ptr / shared_ptr / 裸指针 / 引用
- 能现场说明移动语义在引擎代码中的意义

---

## 第 3 周：内存布局与缓存友好
### 学什么
- alignment / padding
- cache line / locality
- false sharing
- AoS vs SoA
- 数据导向设计基础

### 怎么学
- 写 benchmark 比较不同数据布局的遍历效率
- 用结构体重排实验观察 sizeof 与 padding 变化
- 分析什么样的数据适合连续存储

### 本周任务
- 编写 3 组 AoS / SoA benchmark
- 写一份《缓存友好数据布局实验报告》
- 设计一个适合 ECS 风格处理的数据结构草图

### 本周验收
- 能解释为什么“快”不是抽象概念，而是缓存命中、连续访问和分支行为的结果

---

## 第 4 周：Allocator 基础一
### 学什么
- 线性分配器
- 栈分配器
- freelist 思想
- 内存块管理与 reset 语义

### 怎么学
- 先画内存块示意图，再编码
- 每个 allocator 配单元测试、越界测试和压力测试
- 思考每个 allocator 的适用场景

### 本周任务
- 实现 `LinearAllocator`
- 实现 `StackAllocator`
- 支持对齐分配
- 编写简单性能对比测试

### 本周验收
- 能解释线性分配器为什么适合 frame allocator 场景
- 能解释栈式释放的限制和优势

---

## 第 5 周：Allocator 基础二
### 学什么
- Pool allocator
- free list 复用
- 小对象分配开销
- 内存碎片基础认知

### 怎么学
- 编写固定大小对象池
- 用对象频繁创建/销毁做基准测试
- 对比 `new/delete` 与自定义池的差异

### 本周任务
- 实现 `PoolAllocator`
- 为多个对象类型建立测试样例
- 写 allocator benchmark 总结

### 本周验收
- 能说明 pool allocator 适合什么类型对象
- 能解释固定块大小与灵活性的权衡

---

## 第 6 周：线程与并发基础
### 学什么
- `std::thread`
- `mutex`
- `condition_variable`
- `atomic`
- 生产者消费者模型
- 并发常见 bug 类型

### 怎么学
- 从最朴素同步方案开始写，不急于 lock-free
- 查 cppreference 中 atomic 与内存序的基础页面
- 刻意制造死锁、竞态和 lost wakeup 场景

### 本周任务
- 实现 BlockingQueue
- 实现线程安全 Logger
- 写 2 个多线程测试程序

### 本周验收
- 能解释 mutex、condition_variable 和 atomic 的适用边界
- 能解释什么是数据竞争

---

## 第 7 周：Thread Pool
### 学什么
- 任务队列设计
- worker 生命周期管理
- graceful shutdown
- 任务粒度与调度开销

### 怎么学
- 先写最朴素固定线程池
- 再测试不同任务粒度下的吞吐差异
- 总结线程池“不是越复杂越好”的工程经验

### 本周任务
- 实现 ThreadPool v1
- 支持任务提交与关闭
- 增加 Future / 返回值能力（可选）
- 做简单吞吐 benchmark

### 本周验收
- 能说清楚线程池的核心目标是什么
- 能指出任务过小导致调度收益下降的原因

---

## 第 8 周：TaskGraph / Job System 雏形
### 学什么
- DAG 依赖关系
- task dependency
- 并行 for / 分块任务
- 图调度与完成计数

### 怎么学
- 先支持小规模 task graph
- 先保证正确执行顺序，再考虑性能
- 画出依赖图再写代码

### 本周任务
- 实现 JobSystem v1
- 支持简单依赖关系
- 写一个并行 for demo
- 把系统集成到 engine_core 模块中

### 本周验收
- 能解释 Job System 与普通 ThreadPool 的区别
- 能说明为什么依赖图在引擎中重要

---

### 学什么
- AddressSanitizer
- UBSan
- ThreadSanitizer
- gdb / lldb 基础
- Tracy 基础接入

### 怎么学

### 本周任务

### 本周验收
- 能说明“如何验证代码正确”不只靠肉眼和 printf

- 优化前后对比

### 怎么学

- 记录副作用和取舍

- 输出优化前后数据对比
- 写 1 份 case study

### 本周验收

## 第 49 周：论文或技术复现
### 学什么
- 选择一个可控、可展示、可与现有项目结合的技术
- 例如：SSAO、TAA 简化版、Clustered Light Culling、SSR 简化版

### 怎么学
- 优先选有公开视频、博客、参考实现的题目
- 先读二手资料建立结构，再读 paper 或核心实现
- 目标是“简化复现”而不是完整复刻研究成果

### 本周任务
- 选题
- 完成技术阅读和实现拆解
- 集成到现有 renderer 或单独 demo 中

### 本周验收
- 能讲清该技术解决什么问题、核心思路是什么、自己的实现做了哪些简化

# 十二、执行建议

## 1. 用季度思维而不是周思维焦虑
每一周的任务本质上是在为阶段项目服务，不必追求每周都“做很多新东西”，重点是持续推进。

## 2. 每周只抓一个核心难点
不要让同一周同时承担多个全新难点，例如“第一次写线程池 + 第一次学 Vulkan + 第一次做阴影”。应当把学习曲线拆开。

## 3. 所有项目都要可讲解
你做的不是课程作业，而是求职项目。每个项目都要能讲 5 分钟、10 分钟、15 分钟三个版本。

## 4. 把踩坑当成成果的一部分
引擎岗很看重定位问题和优化问题的能力。你记录的 bug、抓帧分析和性能报告，本身就是高价值成果。

## 5. 始终围绕“像引擎岗候选人”来判断取舍
当时间不够时，优先做：
- 可展示
- 可验证
- 可解释
- 可与岗位要求直接对应

而不是优先做“看起来最难”的内容。

这才是更接近大厂引擎岗候选人的完整状态。
