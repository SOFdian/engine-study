# 第一周学习计划：环境、仓库与工程骨架搭建

> 依据《52 周引擎岗学习路线》中的“第 1 周：环境、仓库与工程骨架搭建”进行细化，目标是把第一周从“知道要做什么”推进到“每天都能执行、每天都有产出”。

------

## 一、本周目标

本周的核心任务不是追求写出很多功能，而是完成一个**后续 52 周都能持续演进的工程底座**。

围绕原路线中第 1 周的要求，本周需要完成以下 4 个结果：

1. 搭建统一的 monorepo 工程目录。
2. 配置 CMake 基础构建流程，支持 Debug / Release。
3. 接入最基础的 Logger、Assert、单元测试与 benchmark 框架。
4. 产出一份清晰 README，能说明目录结构、构建方式和分层目的。

------

## 二、本周学习原则

### 1. 先跑通，再优化

第一周不要追求“架构设计得特别高级”，重点是先把构建、运行、测试和目录组织跑通。

### 2. 每天都必须有可验证产出

每天结束前至少要留下一个可以检查的结果，例如：

- 新增一个模块目录
- 跑通一次构建
- 写完一个 Logger
- 成功执行一个测试
- 更新一次 README

### 3. 学习方式以“查官方 + 动手验证”为主

这一周不建议重度刷视频，而是采用以下顺序：

1. 先看路线要求，明确今天目标。
2. 查官方资料或权威参考。
3. 自己动手搭工程。
4. 把踩坑和结论写到笔记里。

### 4. 每天都保留复盘

每天最后 20~30 分钟，记录：

- 今天做了什么
- 卡在哪里
- 明天第一件事做什么

------

## 三、建议的每日时间分配

如果你这周可以每天投入 **4~6 小时**，建议按下面分配：

- 1 小时：阅读资料，明确概念
- 2~3 小时：动手搭建和编码
- 0.5~1 小时：调试与验证
- 0.5 小时：写笔记 / 更新 README / 做复盘

如果每天只有 **2~3 小时**，则优先级如下：

1. 工程骨架搭建
2. CMake 构建跑通
3. Logger / Assert
4. 测试接入
5. benchmark 接入
6. README 整理

------

## 四、第一周详细日计划

## Day 1：明确目标，搭建基础开发环境

### 今天学什么

- 第 1 周在全年路线中的位置与作用
- Git 仓库的基本组织方式
- C++ 工程开发所需基础环境
- CMake 的最小工作流

### 今天怎么学

1. 先通读原路线里“第 1 周”的内容，明确这周的最终交付是什么。
2. 检查本机开发环境：
   - 编译器（MSVC / clang / gcc）
   - CMake
   - Git
   - IDE 或编辑器（Visual Studio / CLion / VS Code）
3. 打开 CMake 官方 Tutorial，先理解最小 CMake 项目至少需要什么。
4. 不急着写复杂代码，先用一个最小 `main.cpp` 验证“能编译、能运行”。

### 今天要完成的任务

- 安装并确认以下工具可用：
  - Git
  - CMake
  - 编译器
  - IDE / 编辑器
- 新建仓库，例如：`engine-study` 或 `mini-engine-roadmap`
- 初始化 Git 仓库并提交第一次 commit
- 创建最小目录结构：
  - `engine_core`
  - `math`
  - `renderer_cpu`
  - `renderer_gpu`
  - `tools`
  - `tests`
  - `third_party`（可选）
  - `docs`
- 写一个最简单的 `main.cpp`
- 写最小版根目录 `CMakeLists.txt`
- 成功构建并运行 “Hello Engine”

### 今天的产出物

- 可运行的最小 C++ 工程
- 初始目录结构
- 第一次 Git 提交

### 今天的验收标准

- 终端或 IDE 中可以成功构建并运行程序
- 你能说清楚每个目录为什么存在

### 今天的学习记录建议

写一篇简短笔记：
**《Day 1：我的引擎学习仓库如何初始化》**
内容包括：

- 我的开发环境版本
- 我的仓库目录结构
- 遇到的问题和解决方式

## Day 2：系统学习 CMake 最小工程结构

### 今天学什么

- `project()`、`add_executable()`、`add_library()` 的作用
- 源文件如何组织到 target 中
- 根 CMakeLists 与子目录 CMakeLists 的关系
- Debug / Release 的基本区别

### 今天怎么学

1. 按 CMake 官方 Tutorial 走一遍最核心的前几节。
2. 不要只看示例，要把示例迁移到自己的仓库中。
3. 每学一个命令，就在自己的项目里试一次。
4. 重点理解 target 思维，而不是机械记命令。

### 今天要完成的任务

- 把根目录 CMakeLists 分拆为“根 + 子模块”结构
- 为 `engine_core` 建立一个静态库 target
- 为 `tests` 建立单独目录
- 学会在构建目录外进行 out-of-source build
- 尝试分别构建 Debug 和 Release

### 推荐实践步骤

- 根目录只做全局配置和添加子目录
- 模块内各自维护自己的 `CMakeLists.txt`
- 把 `main.cpp` 放到一个 app 或 demo target 中

### 今天的产出物

- 更规范的 CMake 项目结构
- 至少 1 个 library target 和 1 个 executable target
- 成功跑通 Debug / Release 构建

### 今天的验收标准

- 你能解释：
  - 为什么大型项目不把所有源文件都写在一个 `CMakeLists.txt` 里
  - 为什么推荐 out-of-source build
  - Debug 和 Release 的用途差别是什么

### 今天的学习记录建议

写一篇笔记：
**《Day 2：我如何把最小 CMake 工程拆成多模块结构》**

------

## Day 3：工程分层设计与模块边界初步建立

### 今天学什么

- 什么是模块分层
- 为什么引擎工程需要分模块组织
- `engine_core`、`math`、`renderer_cpu` 等目录各自承担什么职责
- include 与 source 的基本组织习惯

### 今天怎么学

1. 先根据原路线给出的目录建议，自己写一版模块职责说明。
2. 不要追求“未来 100% 正确”，重点是先形成一个能迭代的结构。
3. 参考一些中小型开源 C++ 项目，观察别人是如何分模块的。
4. 画一个简单模块关系图。

### 今天要完成的任务

- 在 `docs/` 中写一份模块说明文档
- 为每个模块建立基础文件结构，例如：
  - `include/`
  - `src/`
- 在 `engine_core` 中加入一个最简单的头文件与实现文件
- 在主程序里链接 `engine_core`
- 画出第一版模块依赖图

### 今天的产出物

- 模块职责说明
- 初步 include/src 结构
- 一张简单模块关系图

### 今天的验收标准

- 你能说清楚：
  - 为什么 `math` 不应该直接依赖 `renderer_gpu`
  - 为什么核心基础模块应该尽量保持依赖简单

### 今天的学习记录建议

写一篇笔记：
**《Day 3：我的 mini engine 工程应该如何分层》**

## Day 4：实现 Logger 基础版本

### 今天学什么

- 日志系统在工程中的作用
- 日志等级（Info / Warning / Error / Debug）
- 宏与函数封装的基本思路
- 为什么日志是调试基础设施的一部分

### 今天怎么学

1. 先不要设计复杂日志系统，只做“够用版本”。
2. 先明确日志最少要解决什么问题：
   - 输出信息
   - 标识等级
   - 辅助定位问题
3. 先写一个简单接口，再逐步增加格式与封装。
4. 在主程序中实际调用 Logger，观察输出效果。

### 今天要完成的任务

- 在 `engine_core` 中实现 Logger v1
- 至少支持：
  - `LOG_INFO`
  - `LOG_WARN`
  - `LOG_ERROR`
- 输出格式中至少包含：
  - 日志等级
  - 文本内容
- 在 `main.cpp` 中调用日志接口
- 记录 Logger 的设计思路

### 今天的产出物

- 可用的 Logger v1
- 演示日志输出的示例程序

### 今天的验收标准

- 程序运行时能清晰输出不同级别日志
- 你能解释为什么日志系统不能只靠 `std::cout`

### 今天的学习记录建议

写一篇笔记：
**《Day 4：从 0 写一个最小 Logger》**
重点写：

- 为什么日志要分级
- 为什么这周只做最小版本
- 后续可能怎样扩展

## Day 5：实现 Assert 与错误定位基础设施

### 今天学什么

- Assert 的作用
- Debug 阶段为什么需要主动暴露错误
- 日志与断言的配合关系
- 基础错误定位意识

### 今天怎么学

1. 先理解：Assert 的目标不是“防所有错误”，而是帮助你更快发现不该发生的情况。
2. 先做宏版本，不必一开始就做复杂封装。
3. 故意制造错误，让断言触发一次，观察效果。
4. 思考哪些条件适合写成断言。

### 今天要完成的任务

- 实现 `ASSERT(condition, message)` 宏
- 断言失败时输出足够信息，例如：
  - 文件名
  - 行号
  - 错误信息
- 让 Assert 与 Logger 配合使用
- 在 demo 中故意触发一次断言，验证效果
- 记录哪些场景适合用 Assert，哪些不适合

### 今天的产出物

- Assert 宏
- 一个故意触发断言的 demo
- 一份“日志 + 断言”配合说明

### 今天的验收标准

- 断言失败时能快速定位到问题位置
- 你能解释：
  - Assert 和正常错误处理的区别
  - 为什么引擎代码里常常大量使用断言

### 今天的学习记录建议

写一篇笔记：
**《Day 5：Assert 不是报错替代品，而是调试加速器》**

## Day 6：接入单元测试与 benchmark 框架

### 今天学什么

- 为什么第一周就要接测试
- 单元测试的最小实践方式
- benchmark 的意义
- “能运行”和“可验证”之间的区别

### 今天怎么学

1. 选择一个轻量测试框架和 benchmark 框架。
2. 不要求把所有模块都覆盖，重点是把流程跑通。
3. 先给 Logger、Assert 或一个简单工具函数写 1~2 个测试。
4. benchmark 先测一个很简单的函数，只为熟悉流程。

### 今天要完成的任务

- 接入测试框架
- 在 `tests/` 中写至少 2 个测试用例
- 保证测试可通过 CMake 运行
- 接入 benchmark 框架
- 写一个最简单 benchmark 示例

### 今天的产出物

- 测试 target
- 至少 2 个单元测试
- 至少 1 个 benchmark 示例

### 今天的验收标准

- 你可以一键构建并运行测试
- 你能解释：
  - 为什么测试不是“后面再补”更合适
  - benchmark 在引擎学习里为什么值得尽早接入

### 今天的学习记录建议

写一篇笔记：
**《Day 6：让我的工程第一次具备可测试性》**

## 五、本周推荐的学习材料与使用方式

## 1. CMake 官方 Tutorial

### 学习目标

不是背命令，而是理解最小工程如何逐步演进成多模块工程。

### 使用方式

- Day 1~2 为主线资料
- 每学到一个命令就在自己项目里实践一次
- 把“为什么这么写”记录到笔记里

## 2. cppreference

### 学习目标

查 C++ 基础语法、标准库接口和宏相关用法。

### 使用方式

- 不要从头到尾读
- 遇到问题时按需查询

### 学习目标

掌握最基本的提交节奏和仓库整理习惯。

### 使用方式

- 本周至少完成 5~7 次 commit

### 学习目标

先完成接入，再逐步理解更完整功能。

### 使用方式

- 只读最小上手部分
- 重点是让测试与 benchmark 真正跑起来

完成这一周后，你不一定已经“很懂引擎”，但你应该已经具备以下意识：

## 九、给你的执行建议

这一周最容易出现的问题有两个：

### 1. 过度追求“完美架构”

第一周最重要的是建立骨架，而不是设计一套未来 100% 不会改的结构。只要分层基本合理、构建可运行，就已经达标。

### 2. 只搭环境，不做记录

你以后找工作时，真正能体现成长的，不只是代码，还有：

- README
- 架构图
- 学习笔记
- 踩坑记录

所以第一周就要养成“边做边留档”的习惯。
