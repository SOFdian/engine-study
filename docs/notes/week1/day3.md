### 1. `#pragma once` 是什么意思？

这是**头文件卫士（Header Guard）**的一种现代简写方式。

- **核心作用**：确保在一次编译过程中，同一个头文件只被包含（include）一次。
- **为什么要它**：在复杂的引擎工程中，`A.h` 可能引用了 `Logger.h`，而 `B.h` 也引用了 `Logger.h`。如果 `main.cpp` 同时引用了 A 和 B，如果没有这个指令，编译器会看到两个 `Logger` 的定义，从而报错“重复定义”。
- **对比**：传统的做法是使用宏定义（`#ifndef HEADER_H ...`），但 `#pragma once` 更简洁，且现代编译器（MSVC, Clang, GCC）都已完美支持。

### 2. `namespace` 的意义

`namespace`（命名空间）可以理解为给代码加一个**逻辑前缀**。

- **解决命名冲突**：假设你引用了一个第三方库，里面也有个函数叫 `Log()`，而你的引擎也有个 `Log()`。如果没有命名空间，编译器会不知道调用哪一个。
- **组织逻辑**：通过 `namespace core { ... }`，你告诉其他人：“这是核心模块的功能”。调用时必须写成 `core::LogHelloWorld()`，这就像在身份证号前加上省份代码，确保全局唯一性。

### 3. 如何确定 `#include "core/Logger.h"` 的路径？

这正是 **CMake** 施展魔法的地方。路径的确定遵循以下逻辑：

1. **物理路径**：你的文件实际存在于 `engine_core/include/core/Logger.h`。
2. **配置路径**：在 `engine_core/CMakeLists.txt` 中，你写了 `target_include_directories(engine_core PUBLIC include)`。
3. **查找逻辑**：这行配置告诉编译器：“当有人编译或使用 `engine_core` 时，请把 `include/` 文件夹作为一个搜索根目录”。
4. **相对路径**：既然 `include/` 是根目录，那么从根目录出发找到 `Logger.h` 的路径自然就是 `core/Logger.h`。

### 4. 为什么要用 `STATIC`？

在 `add_library(engine_core STATIC ...)` 中，`STATIC` 代表**静态库**。

- **物理形式**：在 Windows 上生成 `.lib` 文件，在 Linux 上生成 `.a` 文件。
- **链接方式**：当你编译测试程序时，静态库的代码会被直接“拷贝”并嵌入到最终的可执行文件中。
- **为什么适合引擎起步**：
  - **性能**：编译器可以对整个可执行文件进行跨模块优化。
  - **简单**：不需要处理动态库（DLL）在运行时找不到路径的问题。
  - **可靠**：引擎的基础组件（如数学库、核心库）通常非常稳定，适合打包在一起。

## 1. 静态组织结构：文件的“名片”与“内脏”

在目前的 Monorepo 结构中，每个模块都遵循“公开接口与私有实现分离”的原则。

| **文件类型**        | **存放位置**                | **角色与功能**                                               |
| ------------------- | --------------------------- | ------------------------------------------------------------ |
| **头文件 (`.h`)**   | `模块/include/目录/`        | **合同/声明**：定义函数名、参数和命名空间。它是暴露给其他模块的“使用说明书”。 |
| **源文件 (`.cpp`)** | `模块/src/`                 | **执行/实现**：具体的逻辑代码。它是隐藏在内部的“内脏”，不被其他模块直接读取。 |
| **子 CMakeLists**   | `模块/CMakeLists.txt`       | **模块管家**：定义如何将当前目录的代码打包成一个 Target（如库或可执行文件）。 |
| **根 CMakeLists**   | `项目根目录/CMakeLists.txt` | **总指挥**：定义全局编译环境（C++ 标准、Debug/Release），并编排模块的加载顺序。 |

## 2. 构建逻辑层次：CMake 的属性传递

CMake 的核心任务是维护一张**依赖图谱**。它通过“Target（目标）”的概念来管理编译属性。

1. **根目录 `CMakeLists.txt`**：使用 `add_subdirectory()` 递归进入子目录。它首先加载底层的 `engine_core`，确保库定义先于使用。
2. **`engine_core` 模块**：
   - 通过 `add_library(engine_core STATIC)` 声明一个名为 `engine_core` 的静态库目标。
   - 通过 `target_include_directories(... PUBLIC include)` 将 `include` 目录标记为“公共头文件路径”。这意味着任何链接该目标的程序都能自动找到它里面的头文件。
3. **`tests` 模块**：
   - 通过 `add_executable(engine_test)` 声明最终的程序。
   - 通过 `target_link_libraries(engine_test PRIVATE engine_core)` 建立依赖。这一步最关键：它让 `tests` 获得了 `engine_core` 的二进制代码，同时也获得了它的头文件搜索路径。

------

## 3. 编译与链接流程：从源码到二进制

当你点击“构建”时，计算机内部经历了以下自动化步骤：

### 阶段一：预处理与编译（针对每个 `.cpp`）

编译器会独立处理每一个 `.cpp` 文件。

- **处理 `main.cpp`**：看到 `#include <core/Logger.h>`，编译器根据 CMake 提供的路径找到该头文件，将其内容拷贝进 `main.cpp`。只要语法正确，编译器就会生成 `main.obj`。此时它知道要调用 `LogHelloWorld`，但不知道这个函数具体在内存的哪个位置。
- **处理 `Logger.cpp`**：编译器将其编译为 `Logger.obj`（在静态库中会被打包进 `engine_core.lib`）。这里包含了函数的真实机器指令。

### 阶段二：链接（针对最终 `Executable`）

链接器（Linker）出场，它负责“对暗号”。

- 它发现 `main.obj` 有一个对 `LogHelloWorld` 的未解析引用。
- 它在 `engine_core.lib` 中找到了对应的符号定义。
- 链接器将两者结合，计算出正确的跳转地址，最后生成 `engine_test.exe`。

------

### 总结

- **`.h`** 建立了模块间的**沟通语言**。
- **`.cpp`** 承载了模块的**实际劳动力**。
- **子 `CMakeLists.txt`** 封装了**局部属性**。
- **根 `CMakeLists.txt`** 完成了**全局组装**。

这种组织方式确保了当你修改 `Logger.cpp` 的内部实现时，只要 `Logger.h` 的接口没变，`tests` 模块的代码就不需要做任何改动，只需要重新链接即可。这是大型软件工程能够持续迭代的基础。
